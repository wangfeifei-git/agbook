# 小说写作 AI Agent 工具方案

## 1. 项目目标

打造一个**本地优先**的小说写作 AI Agent 工具，帮助作者完成从设定整理、大纲规划、章节编排、正文生成到一致性审核的完整创作流程。

该工具的核心目标不是单纯“生成文本”，而是作为一个围绕长篇小说创作流程运转的智能工作台，提升以下能力：

- 管理复杂设定，避免人物、世界观、时间线混乱
- 规划长篇结构，支持总纲到章节级拆解
- 批量编排章节生成任务，提高创作效率
- 提供审核与修订机制，减少前后矛盾和文风失控
- 支持多模型接入，降低对单一服务商的依赖

---

## 2. 产品定位

### 2.1 产品形态
采用**本地安装模式**，可选实现形态：

- 桌面客户端（推荐）
- 本地 Web 服务 + 浏览器访问

更推荐采用桌面客户端形态，使产品更接近 Cursor 这类“工作台式”体验。

### 2.2 产品定位描述
这是一个面向中长篇小说创作的本地 AI 工作台，围绕“小说资产管理 + AI 工作流编排 + 一致性审核”展开，而不是通用聊天工具。

### 2.3 目标用户
建议优先聚焦以下用户之一，避免初期产品定位过泛：

- 网文作者
- 长篇小说创作者
- 轻小说 / 剧情向内容作者
- 编剧 / 故事策划型创作者

---

## 3. 核心需求分析

## 3.1 本地安装模式

### 需求
避免复杂的服务部署方式，优先采用本地安装使用模式。

### 可行性
高。技术实现成熟，可采用 Electron 或 Tauri。

### 优点
- 用户安装后可直接使用，门槛低
- 本地读写文件更方便，适合小说工程化管理
- 数据可本地保存，隐私性更好
- 更适合沉淀“项目”“设定库”“章节历史”等结构化资产

### 缺点
- 跨平台打包、自动更新、日志排障成本较高
- 本地数据库和配置管理需要额外设计
- 若要兼容插件或复杂扩展，架构需提前预留

### 建议补充
需明确：
- 是否必须支持 macOS / Windows / Linux
- 是否支持离线使用
- 数据保存方式是本地文件、SQLite，还是二者结合
- 是否需要导出为 `md / txt / docx`

---

## 3.2 多模型接入与协议兼容

### 需求
支持通过统一协议接入不同渠道的模型服务。

### 建议表述
建议将需求明确为：

**兼容 OpenAI-style API，支持配置不同模型服务商与代理渠道。**

### 可行性
高。多数模型平台都会提供一定程度的 OpenAI 兼容接口。

### 优点
- 模型来源灵活，不绑定单一厂商
- 可根据场景切换模型，例如大纲、润色、审稿分别使用不同模型
- 便于后续扩展本地模型、私有模型、第三方代理平台

### 风险
- “OpenAI 兼容”通常不是完全兼容
- 流式输出、工具调用、结构化输出可能存在差异
- 不同模型在长文本写作质量上差异很大，需要能力分层

### 建议补充
模型接入层建议支持：
- Base URL
- API Key
- Model Name
- 自定义请求头
- Streaming
- 工具调用（如后期需要）
- 结构化输出（JSON Schema）

同时建议建立模型标签体系，例如：
- 长文本能力强
- 中文写作更稳定
- 成本低
- 适合润色
- 适合规划
- 支持工具调用

---

## 3.3 界面风格方向

### 需求
页面风格参考 Cursor、Antigravity 一类产品。

### 可行性
高，适合“编辑器式工作台”定位。

### 推荐设计方向
整体采用深色、专业、沉浸式风格，布局建议如下：

- 左侧：项目导航、设定、大纲、章节树
- 中间：主要编辑区
- 右侧：AI 助手、上下文注入、审核结果
- 顶部：项目切换、模型选择、任务状态
- 底部：生成日志、任务进度、错误提示

### 优点
- 信息层次清晰
- 符合“长期创作工作台”习惯
- 易于容纳多面板、多阶段创作流程

### 风险
- 如果过于 IDE 化，可能对非技术写作者不够友好
- 需要平衡“管理感”和“沉浸写作感”

### 建议补充
建议支持两种模式：
- 工作台模式：用于管理设定、大纲、章节任务
- 专注写作模式：用于正文创作、润色、修改

---

## 4. 功能规划建议

## 4.1 小说管理

### 功能目标
管理一本小说的基础资料和长期稳定设定。

### 建议包含内容
- 小说基本信息：书名、类型、题材、目标字数、风格
- 世界观设定
- 人物设定
- 势力 / 地点 / 道具设定
- 时间线
- 关键规则与禁区
- 写作风格要求

### 关键要求
- 设定需要支持分类与引用
- 设定需要支持版本管理
- 章节生成时应按需注入相关设定，而不是全部塞入上下文

---

## 4.2 大纲管理

### 功能目标
支持多层级结构化规划，形成从总纲到章纲的清晰链路。

### 建议层级
- 小说总纲
- 卷纲 / 篇章纲
- 章节摘要
- 场景清单（可选）

### 推荐能力
- 从总纲自动拆解为卷纲、章纲
- 支持人工编辑与 AI 辅助扩写
- 标记伏笔、冲突、反转、角色成长
- 支持大纲变更影响提示

### 风险
如果没有层级设计，后续章节生成会严重依赖大段手工描述，影响效率和一致性。

---

## 4.3 章节生成计划编排

### 功能目标
把“生成章节”从单次操作升级为可计划、可执行、可回溯的任务流。

### 建议支持内容
- 指定生成章节范围，例如第 11-20 章
- 每章目标字数
- 每章字数下限 / 上限
- 每章剧情目标
- 生成前是否先产出章纲
- 生成顺序
- 章节生成规则配置
- 是否启用审核
- 审核失败后的处理方式

### 建议将“章节生成规则”单独结构化
不要只把规则写在备注里，建议作为章节计划中的显式配置项，至少包含：

- 字数规则：目标字数、最低字数、最高字数
- 结构规则：是否必须先产出章纲、是否要求分场景生成
- 内容规则：本章必须出现的剧情点、不得出现的内容
- 风格规则：文风偏好、叙事视角、对话密度、描写比例
- 连续性规则：必须衔接的前情、禁止突破的设定边界

### 推荐任务流
1. 读取小说设定与相关上下文
2. 生成章纲
3. 生成场景拆分（可选）
4. 生成正文初稿
5. 执行审核规则
6. 给出修订建议或自动修订
7. 保存版本结果

### 建议区分两种模式
- 批量模式：适合连续编排生成多章
- 单章精修模式：适合重点打磨单章质量

---

## 4.4 审核规则系统

### 功能目标
保证生成内容符合设定、剧情逻辑和风格要求。

### 建议规则类型

#### 硬规则
- 不得违反人物设定
- 不得与时间线冲突
- 不得泄露未到章节的信息
- 必须满足字数区间
- 必须包含指定剧情节点
- 必须满足章节生成规则中的强约束项

#### 软规则
- 文风一致
- 节奏合理
- 信息密度符合章节目标
- 对话和描写比例协调
- 叙事节奏符合本章预设规则

#### 内容质量规则
- 避免重复表达
- 避免角色行为失真
- 避免因果断裂
- 避免情节推进过快或过慢

### 建议输出形式
审核结果不要只做文本提示，建议结构化输出：
- 是否通过
- 问题类型
- 问题描述
- 严重程度
- 修订建议
- 是否允许自动重写

---

## 5. 当前需求的主要优点

- 产品方向明确，不是泛聊天工具
- 本地优先适合长期创作资产管理
- 多模型兼容降低平台绑定风险
- “章节编排 + 审核规则”具备明显差异化
- 适合做成长期使用的创作工作台，而非单次生成器

---

## 6. 当前需求仍需补足的关键点

## 6.1 上下文管理机制
这是小说类 AI 工具的核心难点之一。

不能把所有设定和全文上下文无差别送给模型，否则会导致：
- 成本高
- 响应慢
- 模型注意力分散
- 输出稳定性变差

### 建议补充能力
- 按章节目标检索相关设定
- 自动引用涉及人物、地点、事件
- 只注入必要上下文
- 提供用户可见的“本次注入内容预览”

---

## 6.2 一致性维护机制
长篇写作最怕设定冲突和人物失真。

### 建议补充能力
- 角色状态记录
- 时间线校验
- 已发生事件记录
- 伏笔与回收点标记
- 关键设定冲突提醒

---

## 6.3 版本管理
大纲、设定、章节都可能不断变化。

### 建议补充能力
- 查看历史版本
- 版本差异比较
- 大纲变更后提示受影响章节
- 支持回退到历史草稿

---

## 6.4 Agent 工作流定义
需要明确 Agent 不是“一个聊天框”，而是一套能执行步骤的系统。

### 建议区分几类 Agent
- 设定整理 Agent
- 大纲规划 Agent
- 章节生成 Agent
- 审核 Agent
- 修订 Agent

也可以在 MVP 阶段只保留一个统一 Agent，但底层按照不同任务模板执行。

---

## 7. 推荐的 MVP 范围

为了控制复杂度，第一版建议聚焦以下能力：

### 必做
- 小说项目创建
- 设定管理
- 多级大纲管理
- 单章 / 多章生成计划
- 正文生成
- 审核规则执行
- 模型配置管理（OpenAI 兼容）

### 暂缓
- 多人协作
- 插件系统
- 本地模型复杂适配
- 自动伏笔追踪
- 高级数据分析与写作统计
- 云同步

---

## 8. 技术方案建议

## 8.1 推荐架构
- 前端：React
- UI：Tailwind CSS + 组件库
- 客户端：Tauri（优先）或 Electron
- 本地服务层：Node.js
- 存储：SQLite + 本地 Markdown / JSON
- 模型接入层：统一 OpenAI-compatible Provider
- 工作流引擎：任务式流程编排

## 8.2 数据组织建议
一个小说项目建议包含以下核心对象：
- Novel
- Setting
- Character
- Outline
- ChapterPlan
- ChapterDraft
- ReviewResult
- PromptTemplate
- ModelProfile

## 10. 关键风险与应对

## 10.1 风险：模型输出不稳定
### 应对
- 固定任务模板
- 引入结构化中间产物
- 增加审核与修订闭环

## 10.2 风险：长篇设定越来越混乱
### 应对
- 强制结构化设定
- 做上下文筛选与引用
- 增加一致性检查

## 10.3 风险：产品做成“聊天工具套壳”
### 应对
- 重点建设项目管理、章节编排、审核系统
- 强化工作流，而不是单轮问答体验

## 10.4 风险：功能过多导致首版过重
### 应对
- 明确 MVP
- 优先做单人、本地、长篇创作主链路
- 暂缓协作、插件、云同步等复杂功能

---

## 11. 建议的后续讨论方向

下一步建议优先明确以下问题：

1. 目标用户更偏网文作者，还是偏严肃文学作者？
2. 首版更偏“批量产出”，还是“高质量精修”？
3. 审核规则是固定模板为主，还是支持用户自由配置？
4. 是否需要支持本地模型？
5. 是否需要导出到外部写作工具格式？

---

## 12. 结论

该方案整体可行，且具备明确的产品差异化潜力。

真正有价值的部分，不是“接一个模型然后生成一段小说”，而是建立一个本地运行、支持多模型、围绕小说工程化创作流程运转的 Agent 工作台。  
如果产品设计得当，它的核心竞争力将来自：

- 结构化创作资产管理
- 长篇上下文与一致性维护
- 章节计划编排
- 审核与修订闭环
- 本地优先的稳定体验

这个方向适合先做一个聚焦的 MVP，再逐步扩展到更完整的小说创作平台。
